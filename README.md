# Sanjeevani

Speak or type a health question in any major Indian language. Sanjeevani
transcribes it, translates it to English, and gives you a plain-language
answer — clearly marked as a draft until a verified clinical source is
connected.

```
sanjeevani/
├── website/            All website content
│   ├── frontend/        static HTML/CSS/JS UI
│   └── backend/         FastAPI server (serves the UI + the API)
├── model/               All model content
│   ├── asr/              AI4Bharat IndicConformer — speech -> native-language text
│   ├── translation/      AI4Bharat IndicTrans2 — native-language text -> English
│   ├── llm/               Groq-backed answer generation + a placeholder knowledge hook
│   ├── pipeline.py        ties the three stages together
│   └── config.py          all settings, read from environment variables
├── requirements.txt
├── .env.example
└── README.md
```

## What's real here, and what's a placeholder

- **Speech recognition and translation are real, pretrained AI4Bharat models** —
  nothing about them is mocked.
- **The answer text is generated by an LLM via Groq**, using your API key.
- **The medical grounding is an intentional placeholder.** `model/llm/knowledge_base.py`
  always returns "not grounded," which keeps the system prompt extra-cautious and
  keeps the frontend's "DRAFT — pending clinical source" stamp honest. When you have
  a real medical data source to connect, that's the one file to edit — see the
  docstring in `knowledge_base.py`.

## 1. Install

```bash
cd sanjeevani
python3.11 -m venv .venv && source .venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt
```

**System dependency:** browser-recorded audio typically arrives as WebM/Opus.
`torchaudio` needs an `ffmpeg` install on your system PATH to decode it
(`apt install ffmpeg` / `brew install ffmpeg`). Uploaded `.wav`/`.mp3`/`.flac`/`.ogg`
files work either way.

## 2. Add your Groq API key

```bash
cp .env.example .env
# then edit .env and set:
# GROQ_API_KEY=your-key-here
```

Get a key at [console.groq.com/keys](https://console.groq.com/keys). Nothing in this
codebase hardcodes a key — it's read from the environment at request time.

## 3. Hugging Face access (for the AI4Bharat models)

```bash
huggingface-cli login
```

`ai4bharat/indic-conformer-600m-multilingual` and the IndicTrans2 checkpoint both
download automatically on first use and are cached locally afterwards.

## 4. Run it

```bash
uvicorn website.backend.main:app --reload --port 8000
```

Open **http://localhost:8000**. The first request that actually uses a model
(ASR, translation, or the LLM) will be slow while that model loads and caches —
everything after that is fast.

## How a request flows

1. You speak (recorded in-browser via `MediaRecorder`) or type, and pick a language
   or leave it on "Auto-detect."
2. **Speech path:** `model/asr/indic_asr.py` transcribes with IndicConformer in the
   selected language, or — if "Auto-detect" — tries a short candidate list and picks
   the most speech-plausible result (see the caveat below).
   **Text path:** language is detected deterministically from the Unicode script the
   typed text uses (a real, reliable technique for text, unlike audio).
3. `model/translation/indic_translator.py` translates the result to English with
   IndicTrans2 (skipped if the input was already English).
4. `model/llm/knowledge_base.py` is checked for grounding context (currently always
   empty — the placeholder), and `model/llm/groq_client.py` drafts an answer with Groq,
   with safety instructions baked into the system prompt (no diagnoses, always
   recommend a real health worker, flag emergencies and India's 108 ambulance number).
5. The website renders the transcript, the English translation, and the answer.

## A known limitation: audio language auto-detection

IndicConformer doesn't have a built-in "detect the language" mode — you have to tell
it which language to decode. `detect_and_transcribe()` in `model/asr/indic_asr.py`
works around this by decoding the same clip against a handful of candidate languages
and picking the one whose output looks most like plausible speech (word-rate
heuristics). This is noticeably weaker than a dedicated language-ID model. For
reliable results, prefer having the person pick their language from the dropdown
(the default UI behavior is "Auto-detect," but nudging users to pick explicitly will
give better transcripts). For a stronger fix, look at AI4Bharat's dedicated **IndicLID**
model as a drop-in replacement for the heuristic.

## Extending this

- **Connect a real medical source:** edit `model/llm/knowledge_base.py::retrieve_context`.
- **Add more languages:** extend `SUPPORTED_LANGUAGES` in `model/config.py` (needs both
  an IndicConformer language code and an IndicTrans2 FLORES code — check each model's
  card for the full list of the 22 supported languages).
- **Swap the LLM:** `model/llm/groq_client.py` is the only file that talks to Groq;
  point it at a different OpenAI-compatible endpoint if needed.
